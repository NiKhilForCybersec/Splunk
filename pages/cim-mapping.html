CIM Mapping Guide for Detection Engineering

Map your data to CIM data models for tstats performance and ES compatibility

Why CIM Mapping Matters

ℹ️ The Performance Difference

Without CIM: index=akamai | stats count by src → Scans raw events, slow

With CIM: | tstats count FROM datamodel=Web BY Web.src → Uses accelerated summary, 10-100x faster

Benefits of CIM Compliance

  • Performance: tstats against accelerated data models is orders of magnitude faster
  • Portability: Same SPL works across different data sources
  • ES Integration: ES correlation searches expect CIM field names
  • Dashboard Compatibility: ES dashboards use CIM fields
  • Vendor Consistency: Compare Akamai, Cloudflare, F5 with same query

Akamai WAF → CIM Web Data Model

Field Mapping Table

CIM Field Akamai Field Mapping Method Priority
src attackData.clientIP FIELDALIAS or TA extraction Required
dest httpMessage.host FIELDALIAS or TA extraction Required
action attackData.configId or rule action EVAL: allowed/blocked based on response Required
url httpMessage.path + httpMessage.query EVAL concatenation Recommended
http_method httpMessage.method FIELDALIAS Recommended
status httpMessage.status FIELDALIAS Recommended
http_user_agent httpMessage.requestHeaders.User-Agent Extraction + FIELDALIAS Recommended
vendor — EVAL: "Akamai" Optional
product — EVAL: "Kona Site Defender" or "App & API Protector" Optional
signature attackData.rules[].ruleId MV extraction Optional
category attackData.rules[].ruleTag MV extraction Optional

props.conf Configuration

# $SPLUNK_HOME/etc/apps/TA-akamai/local/props.conf

[akamai:siem]
# Field aliases for CIM Web
FIELDALIAS-src = attackData.clientIP AS src
FIELDALIAS-dest = httpMessage.host AS dest
FIELDALIAS-http_method = httpMessage.method AS http_method
FIELDALIAS-status = httpMessage.status AS status
FIELDALIAS-url_path = httpMessage.path AS url_path
FIELDALIAS-url_query = httpMessage.query AS url_query

# Calculated fields
EVAL-url = if(isnotnull(url_query), url_path."/".url_query, url_path)
EVAL-action = case(
    match(attackData.policyActions, "deny"), "blocked",
    match(attackData.policyActions, "alert"), "allowed",
    isnotnull(httpMessage.status) AND httpMessage.status>=400, "blocked",
    1=1, "allowed"
)
EVAL-vendor = "Akamai"
EVAL-product = "Web Application Firewall"

# Tag for data model
LOOKUP-cim_web = cim_web_action_lookup inputField OUTPUT outputField

tags.conf Configuration

# $SPLUNK_HOME/etc/apps/TA-akamai/local/tags.conf

[source::*akamai*]
web = enabled

[sourcetype::akamai:siem]
web = enabled

# OR use eventtypes:
# [eventtype=akamai_waf_events]
# web = enabled

Validating CIM Compliance

Step 1: Verify Field Extraction

# Check if CIM fields are extracted
index=akamai sourcetype=akamai:siem earliest=-1h
| head 100
| table _time, src, dest, action, http_method, status, url
| where isnotnull(src) AND isnotnull(dest) AND isnotnull(action)

# Check field population rates
index=akamai sourcetype=akamai:siem earliest=-24h
| stats 
    count as total,
    count(src) as src_count,
    count(dest) as dest_count,
    count(action) as action_count,
    count(http_method) as method_count,
    count(url) as url_count
| eval src_pct = round(src_count/total*100, 1)
| eval dest_pct = round(dest_count/total*100, 1)
| eval action_pct = round(action_count/total*100, 1)

Step 2: Verify Data Model Mapping

# Check if data appears in Web data model
| tstats count FROM datamodel=Web WHERE index=akamai BY Web.src, Web.dest
| head 10

# If this returns results, CIM mapping is working
# If empty, check tags and field mappings

# Verify data model acceleration
| rest /services/admin/summarization by_tstats=t
| search datamodel_name=Web
| table datamodel_name, summary.complete, summary.access_count

Step 3: Validate Action Field Values

# CIM expects action to be: allowed, blocked, deferred
# Check what values you actually have
index=akamai sourcetype=akamai:siem earliest=-24h
| stats count by action
| sort -count

# If you see values like "deny", "alert", "monitor" instead of 
# "allowed"/"blocked", your EVAL-action isn't working correctly

✅ CIM Validation Checklist

  • ☐ src field populated >95% of events
  • ☐ dest field populated >95% of events
  • ☐ action field = "allowed" or "blocked"
  • ☐ | tstats count FROM datamodel=Web WHERE index=akamai returns data
  • ☐ Data model acceleration is enabled and complete

Fallback: When CIM Isn't Possible

Sometimes vendor add-ons don't fully CIM-map, or you can't modify props.conf. Here's what to do:

Option 1: Inline Field Normalization

# Normalize fields in your detection SPL
index=akamai sourcetype=akamai:siem earliest=-5m
| eval src = coalesce('attackData.clientIP', src_ip, client_ip)
| eval dest = coalesce('httpMessage.host', dest_host, server)
| eval action = case(
    match(policyActions, "deny"), "blocked",
    status >= 400, "blocked",
    1=1, "allowed"
)
| stats count by src, dest, action
| where action="blocked" AND count > 100

Option 2: Lookup-Based Normalization

# Create a field normalization lookup
# waf_field_normalization.csv:
# raw_action,cim_action
# deny,blocked
# block,blocked
# alert,allowed
# monitor,allowed
# pass,allowed

# Use in SPL:
index=akamai sourcetype=akamai:siem earliest=-5m
| lookup waf_field_normalization.csv raw_action AS policyAction OUTPUT cim_action AS action
| stats count by src, dest, action

Option 3: Summary Index with Normalization

# Create a scheduled search that normalizes and indexes
# Run hourly to create CIM-compliant summary

index=akamai sourcetype=akamai:siem earliest=-65m latest=-5m
| eval src = coalesce('attackData.clientIP', src_ip)
| eval dest = coalesce('httpMessage.host', host)
| eval action = case(match(policyActions, "deny"), "blocked", 1=1, "allowed")
| eval http_method = upper(method)
| stats count as event_count by _time, src, dest, action, http_method
| eval sourcetype = "akamai:siem:normalized"
| collect index=waf_summary

# Then query the summary:
index=waf_summary sourcetype=akamai:siem:normalized
| stats sum(event_count) as count by src, dest, action

⚠️ Trade-offs of Each Approach

ApproachProsCons
Inline normalization No additional config Slower, repeated in every search
Lookup normalization Easy to update mappings Still hits raw data
Summary index Fast queries Data duplication, storage cost

Entra ID / Azure AD Field Mapping

CIM Field Entra Field Notes
user properties.userPrincipalName UPN format
action properties.status.errorCode 0 = success, else failure
src properties.ipAddress Client IP
app properties.appDisplayName Target application
authentication_method properties.authenticationDetails[].authenticationMethod MFA method used

ℹ️ Note on Entra ID Sourcetypes

Sourcetype depends on your ingestion method. Common variants:

  • azure:aad:signin - via Splunk Add-on for MS Cloud Services
  • azure:monitor:aad - via Azure Event Hub
  • mscs:azure:eventhub - some Event Hub configs

Always verify your actual sourcetype: index=azure | stats count by sourcetype
