MFA Detection Use Cases

MFA-specific detections focused on challenge/response anomalies ‚Äî NOT general identity behaviors

üéØ Project Scope: MFA-Specific Detections Only

These detections focus on how MFA challenges are handled, not general login behaviors. All use the same Entra ID Sign-in and Audit logs, but detect different patterns.

‚úì In Scope (MFA):
  • Push bombing / MFA fatigue
  • MFA bypass attempts
  • MFA disabled for users
  • MFA device registration
‚úó Out of Scope (Identity):
  • Impossible travel
  • Risky country login
  • Brute force attacks
  • After-hours login

üö® Priority Order: UC2 ‚Üí UC1 ‚Üí UC4 ‚Üí Rest

UC2 (Breach Indicator) is your MOST IMPORTANT detection. It tells you an attack succeeded. Deploy UC2 first, then UC1, then UC4 (MFA Disabled), then the rest.

Step 1: Validate Data is Flowing

Run These Validation Queries First
``` Entra ID Sign-in Logs - Check MFA events exist
index=azure sourcetype=azure:aad:signin earliest=-24h
| spath path=properties.status.errorCode output=error_code
| where error_code IN (0, 500121, 50074, 50076)
| stats count BY error_code
| eval event_type = case(error_code=0, "Success", error_code=500121, "MFA Denied", error_code=50074, "MFA Incomplete", error_code=50076, "MFA Failed")

``` CyberArk - Check MFA events exist
index=cyberark sourcetype=cyberark:identity:audit earliest=-24h
| where EventType IN ("Cloud.Core.MfaDenied", "Cloud.Core.MfaAuthentication", "Cloud.Core.MfaTimeout")
| stats count BY EventType Result

``` Entra ID Audit Logs - Check for MFA config changes
index=azure sourcetype=azure:aad:audit earliest=-7d
| spath path=properties.operationType output=operation
| search operation IN ("Add", "Update", "Delete") properties.targetResources{}.modifiedProperties{}.displayName="*MFA*" OR properties.targetResources{}.modifiedProperties{}.displayName="*Authentication*"
| stats count

üî¥ Critical Priority ‚Äî Active Attack / Breach Indicators

üö® MFA-UC2: MFA Denials ‚Üí Success (BREACH)

HIGHEST PRIORITY. Multiple MFA denials followed by SUCCESS = attacker likely got user to approve. This is ACTIVE BREACH detection.

Before Steps:
  • Verify error_code 500121 events exist in Sign-in logs
  • Confirm privileged_users.csv is uploaded
  • Validate CyberArk MfaDenied events present
CRITICAL T1621, T1078 Every 5 min

MFA-UC1: MFA Fatigue / Push Bombing

Attacker spams MFA push notifications hoping user accidentally approves. Used in Uber, Cisco, Microsoft breaches.

Before Steps:
  • Count 500121 (MFA Denied) events in last 24h
  • Verify userPrincipalName field extracts
  • Check CyberArk EventType=Cloud.Core.MfaDenied
CRITICAL T1621 Every 5 min

üü† High Priority ‚Äî MFA Security Weakening

MFA-UC3: MFA Satisfied Without Challenge

Authentication succeeds with "MFA satisfied" but no actual challenge was presented to the user. Indicates MFA bypass or misconfiguration.

Before Steps:
  • Verify authenticationRequirement field exists
  • Check mfaDetail object is populated
  • Baseline normal MFA skip scenarios (trusted device)
HIGH T1556.006 Every 15 min

MFA-UC4: MFA Disabled for Admin Account

MFA requirement removed from a privileged user account. Critical security control weakened ‚Äî could indicate insider threat or compromised admin.

Before Steps:
  • Verify Entra ID Audit logs are flowing
  • Confirm privileged_users.csv is current
  • Test query: search for "StrongAuthenticationRequirement"
HIGH T1556 Real-time

üü° Medium Priority ‚Äî MFA Persistence & Evasion

MFA-UC5: New MFA Device Registration

Attacker with account access registers their own authenticator app or security key for persistent access even after password reset.

Before Steps:
  • Verify Audit log operations for device registration
  • Search: "Register security info" or "Add authentication method"
  • Baseline normal device registration patterns
MEDIUM T1098.005 Every 15 min

MFA-UC6: MFA Method Downgrade

User's MFA method changed from strong (FIDO2, Authenticator) to weak (SMS, Voice call). May indicate attacker preparing for SIM swap.

Before Steps:
  • Create mfa_method_strength.csv lookup
  • Verify mfaDetail.authMethod field extracts
  • Baseline current MFA method distribution
MEDIUM T1556 Every 15 min
