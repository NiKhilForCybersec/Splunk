üì• Universal Forwarder (UF)

What it does: Lightweight agent that collects and forwards data

Where it runs: On servers/endpoints you want to collect from

Key points:

  • No parsing/indexing ‚Äî just forwarding
  • ~50MB footprint, minimal CPU
  • Configured via inputs.conf and outputs.conf
  • Can buffer data if connection lost

üì§ Heavy Forwarder (HF)

What it does: Full Splunk instance that can parse/filter before forwarding

Where it runs: Intermediate layer for complex data

Key points:

  • Can parse, filter, mask, route data
  • Higher resource requirements
  • Useful for data transformation
  • Often used for syslog aggregation

üåê HTTP Event Collector (HEC)

What it does: RESTful API endpoint for sending data over HTTP/HTTPS

Where it runs: On indexers/Splunk Cloud

Key points:

  • Token-based authentication
  • Preferred method for cloud services
  • JSON or raw data format
  • Can specify index, sourcetype, host at send time

üíæ Indexer

What it does: Processes, stores, and indexes incoming data

Where it runs: Core of Splunk deployment

Key points:

  • Parses events and extracts timestamps
  • Applies index-time field extractions
  • Compresses and stores in buckets
  • Creates TSIDX files for fast searching

Index Time vs Search Time ‚Äî Critical Concept

‚ö†Ô∏è This Is a Job Interview Question
Understanding the difference between index-time and search-time operations is fundamental. Get this wrong and you'll look like a beginner.
Aspect Index Time Search Time
When it happens When data is ingested/indexed When you run a search
Can be changed? NO ‚Äî once indexed, it's permanent YES ‚Äî can change extractions anytime
Performance impact Slightly slower indexing, faster searches No indexing impact, computed per search
Configuration props.conf with TRANSFORMS props.conf with EXTRACT/REPORT
Use cases Critical fields like host, source, sourcetype Most field extractions
Example Routing events to specific index Extracting IP addresses from logs

Default Indexed Fields (Always Available)

# These fields are ALWAYS indexed: host # The host that generated the event source # The file/input path sourcetype # The data format identifier index # Which index the event is stored in _time # The event timestamp _raw # The original raw event text _indextime # When the event was indexed

Search Flow Step-by-Step

1. User Submits Query

User enters SPL in Search & Reporting app

2. Search Head Parses SPL

Validates syntax, identifies search phases

3. Distributed to Indexers

Search job sent to all indexers in parallel

4. Indexers Search Buckets

Each indexer searches its local buckets + SmartStore

5. Partial Results Returned

Each indexer sends matching events to search head

6. Search Head Merges & Processes

Applies stats, eval, sort, etc. and returns final results

Search Pipeline Phases

1. Parsing SPL is parsed into search phases
2. Bloom Filter Quick elimination of buckets that definitely don't match
3. TSIDX Lookup Index files checked for matching terms
4. Raw Data Scan Actual events retrieved from buckets
5. Field Extraction Search-time extractions applied
6. Filtering WHERE, search terms applied
7. Transforming stats, eval, etc. commands processed

Configuration File Precedence

Why Settings Sometimes Don't Work
Splunk has a complex configuration precedence system. A setting in one location can override another. This is a common source of confusion.

Precedence Order (Highest to Lowest)

  1. User-specific: $SPLUNK_HOME/etc/users/<username>/<app>/local/
  2. App local: $SPLUNK_HOME/etc/apps/<app>/local/
  3. App default: $SPLUNK_HOME/etc/apps/<app>/default/
  4. System local: $SPLUNK_HOME/etc/system/local/
  5. System default: $SPLUNK_HOME/etc/system/default/
Viewing Effective Configuration
Use this command to see what configuration is actually in effect:
| btool props list --debug (command line only)
