WAF-UC4: SQL Injection / XSS Campaign Detection

CRITICAL PRIORITY: A single successful SQLi can compromise your ENTIRE database. A successful XSS can steal admin sessions and credentials. Treat every alert from this detection seriously.

Severity CRITICAL
MITRE T1190
Schedule Every 5 min
Time Range Last 10 min

Step A: Build SPL in Search First

Where: Splunk Search ‚Üí Build and test SPL

A Injection Campaign Detection SPL

index=akamai sourcetype=akamai:siem earliest=-10m
| spath
| eval waf_action=coalesce(action, mvindex('attackData.rules{}.ruleAction',0))
| eval src_ip=coalesce(src_ip,'attackData.clientIP')
| eval signature=coalesce(signature,'attackData.rules{}.ruleTag')
| eval uri=coalesce(uri,'httpMessage.requestUri')
| eval vhost=coalesce(vhost,'httpMessage.requestHeaders.Host')
| where waf_action="deny" OR waf_action="block"
| where signature IN ("SQLI","SQL_INJECTION","XSS","CROSS_SITE_SCRIPTING","CMDI","COMMAND_INJECTION","LFI","LOCAL_FILE_INCLUSION","RFI","REMOTE_FILE_INCLUSION","SSTI","XXE")
| bin _time span=5m
| stats count as attacks 
        dc(src_ip) as unique_attackers 
        dc(signature) as attack_vectors
        values(signature) as attack_types
        values(src_ip) as attacker_ips
        values(uri) as target_uris
        BY _time, vhost
| where attacks > 15
| eval campaign_type = case(
    attack_vectors > 3, "Multi-Vector Campaign",
    unique_attackers > 5, "Distributed Injection Campaign",
    1=1, "Focused Injection Attempt"
)
| eval severity = case(
    attack_vectors > 3, "critical",
    attacks > 100, "critical",
    attacks > 50, "high",
    1=1, "medium"
)
| eval has_sqli = if(match(mvjoin(attack_types,","), "(?i)sql"), "YES", "NO")
| eval has_rce = if(match(mvjoin(attack_types,","), "(?i)(cmd|command|rfi)"), "YES", "NO")
| table _time vhost attacks unique_attackers attack_vectors attack_types has_sqli has_rce campaign_type attacker_ips target_uris severity

CRITICAL: Backend Bypass Verification

‚ö†Ô∏è WAF blocks don't guarantee protection!

Always verify if ANY requests got through the WAF and caused backend errors. A successful bypass means the attacker may have already compromised your database.

! Run This Check Immediately When You See Injection Alerts

index=akamai sourcetype=akamai:siem earliest=-1h vhost="$vhost$"
| spath
| eval waf_action=coalesce(action, mvindex('attackData.rules{}.ruleAction',0))
| eval status=coalesce(status,'httpMessage.status')
| eval signature=coalesce(signature,'attackData.rules{}.ruleTag')
| eval src_ip=coalesce(src_ip,'attackData.clientIP')
| eval uri=coalesce(uri,'httpMessage.requestUri')
| where signature IN ("SQLI","SQL_INJECTION","XSS","CMDI","LFI","RFI")
| stats count as total 
        count(eval(waf_action="allow" OR waf_action="none" OR waf_action="pass")) as allowed_through
        count(eval(status>=500)) as backend_errors
        values(uri) as affected_uris
        values(src_ip) as source_ips
        BY signature
| where allowed_through > 0 OR backend_errors > 0
| eval BYPASS_DETECTED = if(allowed_through > 0, "YES - ESCALATE IMMEDIATELY", "No")
| table signature total allowed_through backend_errors BYPASS_DETECTED affected_uris source_ips
If BYPASS_DETECTED = YES or backend_errors > 0:
  1. Escalate immediately to security lead and IR team
  2. Block the source IPs at network level (firewall, not just WAF)
  3. Check database logs for unauthorized queries or data access
  4. Review affected application logs for successful exploitation
  5. Consider taking application offline if active exploitation confirmed
  6. Engage incident response for full investigation
