WAF-UC1: Block Volume Anomaly Detection

Detect significant spikes in deny/block volume indicating scanning, exploit attempts, or DDoS activity. Uses statistical baseline to identify anomalous traffic patterns.

Default Severity MEDIUM
MITRE ATT&CK T1190
Schedule Every 5 min
Time Range Last 10 min

Step 1: Decision Tree (Detection Logic)

Visual representation of the detection logic flow

flowchart TD
    START[WAF Event Received] --> A{Is waf_action = deny OR block}
    A -->|No| IGNORE[Ignore event - not a block]
    A -->|Yes| B[Aggregate events into 5-minute time buckets]
    B --> C[Count block_count per bucket]
    C --> D[Calculate 24h baseline using eventstats]
    D --> D1[baseline = avg of block_count]
    D1 --> D2[stdev = standard deviation of block_count]
    D2 --> E[Calculate dynamic threshold]
    E --> E1[threshold = baseline + 2 x stdev]
    E1 --> F{Is threshold LT 50}
    F -->|Yes| G[Set threshold = 50 as minimum floor]
    F -->|No| H[Keep calculated threshold]
    G --> I{Is block_count GT threshold}
    H --> I
    I -->|No| NORMAL[Normal traffic volume - no alert]
    I -->|Yes| J{Is block_count GT 10 absolute minimum}
    J -->|No| NORMAL
    J -->|Yes| K[Calculate deviation in sigma units]
    K --> K1[deviation = block_count minus baseline divided by stdev]
    K1 --> L{Is deviation GTE 5 sigma}
    L -->|Yes| CRIT[Set severity = CRITICAL]
    L -->|No| M{Is deviation GTE 3 sigma}
    M -->|Yes| HIGH[Set severity = HIGH]
    M -->|No| MED[Set severity = MEDIUM]
    CRIT --> CREATE[Create Notable Event in ES]
    HIGH --> CREATE
    MED --> CREATE
    CREATE --> THROTTLE{Was Notable created for same condition in last 30 min}
    THROTTLE -->|Yes| SUPPRESS[Suppress duplicate - do not create new Notable]
    THROTTLE -->|No| ALERT[Notable sent to Incident Review queue]
    ALERT --> ENRICH[Enrich with unique_ips and affected hosts]

Step A: Build SPL in Search First

Where: Splunk Search → Build and test SPL until results are stable and accurate before creating correlation search

A1 Variant A: Global Spike (Recommended Start)

Use when: You want a single "something big is happening" alert without per-application granularity. Produces 1 notable per time bucket when threshold exceeded. Best for initial deployment. 

index=akamai sourcetype=akamai:siem earliest=-24h
| spath
| eval waf_action=coalesce(action, mvindex('attackData.rules{}.ruleAction',0))
| where waf_action="deny" OR waf_action="block"
| bin _time span=5m
| eval src_ip=coalesce(src_ip,'attackData.clientIP')
| eval vhost=coalesce(vhost,'httpMessage.requestHeaders.Host')
| stats count as block_count 
        dc(src_ip) as unique_ips 
        values(vhost) as hosts 
        by _time
| eventstats avg(block_count) as baseline stdev(block_count) as stdev
| eval stdev=if(stdev=0,1,stdev)
| eval threshold = baseline + (2*stdev)
| eval threshold = if(threshold < 50, 50, threshold)
| where block_count > threshold AND block_count > 10
| eval deviation=round((block_count-baseline)/stdev,2)
| eval severity=case(deviation>=5,"critical", deviation>=3,"high", 1=1,"medium")
| table _time block_count baseline threshold deviation unique_ips hosts severity

A2 Variant B: Per-Host Spike (More Actionable)

Use when: You want to know "which application is under attack" - more actionable for triage. Produces separate notables per vhost. Better for mature deployments. 

index=akamai sourcetype=akamai:siem earliest=-24h
| spath
| eval waf_action=coalesce(action, mvindex('attackData.rules{}.ruleAction',0))
| where waf_action="deny" OR waf_action="block"
| bin _time span=5m
| eval src_ip=coalesce(src_ip,'attackData.clientIP')
| eval vhost=coalesce(vhost,'httpMessage.requestHeaders.Host')
| stats count as block_count dc(src_ip) as unique_ips by _time vhost
| eventstats avg(block_count) as baseline stdev(block_count) as stdev by vhost
| eval stdev=if(stdev=0,1,stdev)
| eval threshold = baseline + (2*stdev)
| eval threshold = if(threshold < 25, 25, threshold)
| where block_count > threshold AND block_count > 10
| eval deviation=round((block_count-baseline)/stdev,2)
| eval severity=case(deviation>=5,"critical", deviation>=3,"high", 1=1,"medium")
| table _time vhost block_count baseline threshold deviation unique_ips severity

A3 Variant C: 7-Day Baseline (Stable Traffic)

Use when: Traffic has stable weekly patterns (e.g., lower on weekends). More accurate baseline but heavier on Splunk resources. Use after initial deployment is stable. 

index=akamai sourcetype=akamai:siem earliest=-7d
| spath
| eval waf_action=coalesce(action, mvindex('attackData.rules{}.ruleAction',0))
| where waf_action="deny" OR waf_action="block"
| bin _time span=5m
| eval src_ip=coalesce(src_ip,'attackData.clientIP')
| eval vhost=coalesce(vhost,'httpMessage.requestHeaders.Host')
| stats count as block_count dc(src_ip) as unique_ips values(vhost) as hosts by _time
| eventstats avg(block_count) as baseline stdev(block_count) as stdev
| eval stdev=if(stdev=0,1,stdev)
| eval threshold = baseline + (2*stdev)
| eval threshold = if(threshold < 50, 50, threshold)
| where block_count > threshold AND block_count > 10
| eval deviation=round((block_count-baseline)/stdev,2)
| eval severity=case(deviation>=5,"critical", deviation>=3,"high", 1=1,"medium")
| table _time block_count baseline threshold deviation unique_ips hosts severity

Best practice: Start with 24h baseline (Variant A). Once detection is stable for 2 weeks, move to 7d baseline for better accuracy.

Step C: Add Drilldown Search

Where: Add as Adaptive Response Action OR linked search in Notable Event settings

C Drilldown: Show Top Attackers and Targets

This search runs when an analyst clicks into the notable to see who is attacking and what they're targeting:

index=akamai sourcetype=akamai:siem earliest=-15m
| spath
| eval waf_action=coalesce(action, mvindex('attackData.rules{}.ruleAction',0))
| where waf_action="deny" OR waf_action="block"
| eval src_ip=coalesce(src_ip,'attackData.clientIP')
| eval vhost=coalesce(vhost,'httpMessage.requestHeaders.Host')
| eval uri=coalesce(uri,'httpMessage.requestUri')
| eval signature=coalesce(signature,'attackData.rules{}.ruleTag')
| stats count as blocked_requests
        dc(uri) as unique_uris 
        values(uri) as sample_uris 
        values(signature) as attack_types
        by vhost src_ip
| sort - blocked_requests
| head 20

Step D: Validate Notables

Where: ES → Incident Review → Filter by "WAF-UC1"

D Validation Checklist

Check ALL of these before marking detection as "production ready":

  • ✅ Notable appears within 10 minutes of enabling rule (if traffic exists)
  • ✅ All expected fields visible: vhost, block_count, baseline, threshold, deviation, unique_ips, hosts
  • ✅ Severity is correct: deviation ≥5 = critical, ≥3 = high, else medium
  • ✅ Notable is NOT spamming: if >10/hour for same condition, increase throttle or threshold
  • ✅ Drilldown search works: click into notable and verify drilldown returns relevant data
  • ✅ MITRE tagging visible: T1190 shows in Notable details
  • ✅ Historical validation: run search over known attack period to verify it would have detected
  • ✅ False positive check: run over 7 days of normal traffic to estimate FP rate

Step E: Tune Detection

E1 Tuning Parameter Reference

ParameterDefaultFewer AlertsMore CoverageWhat It Controls
Minimum floor 50 100-200 20-30 Prevents alerts on statistically significant but low-volume spikes
Sigma multiplier 2σ 3σ 1.5σ How many standard deviations above baseline triggers alert
Baseline window 24h 7d 12h Longer = more stable baseline, shorter = adapts faster to changes
Time bucket span 5m 10m 2m Smaller = faster detection but more noise, larger = smoother
Throttle duration 30m 60m 15m How long to suppress duplicate alerts for same condition
Schedule frequency 5m 10m 2m How often correlation search runs (affects detection latency)
Absolute minimum 10 25 5 Minimum blocks required regardless of statistics

E2 Additional Tuning SPL Snippets

Allowlist internal/test IPs:

| where NOT cidrmatch("10.0.0.0/8", src_ip) 
        AND NOT cidrmatch("192.168.0.0/16", src_ip)
        AND NOT cidrmatch("172.16.0.0/12", src_ip)

Exclude non-production environments:

| where NOT match(vhost, "(?i)(dev|test|staging|uat|qa|sandbox)")

Business hours only detection:

| eval hour=strftime(_time,"%H") 
| where hour >= 6 AND hour <= 22

Exclude known scanner IPs (using lookup):

| lookup scanner_allowlist.csv src_ip OUTPUT is_scanner 
| where is_scanner!="true" OR isnull(is_scanner)

Separate production vs non-production alerting:

| eval environment = case(
    match(vhost, "(?i)(prod|www\.|api\.)"), "production",
    match(vhost, "(?i)(dev|test|staging)"), "non-production",
    1=1, "unknown"
)
| where environment="production"

Step F: Handle False Positives

F Common False Positives & Solutions

FP SourceHow to IdentifySolution
Internal vulnerability scans Same source IPs every Tuesday night, Nessus/Qualys/Rapid7 user agents in logs Create scanner_allowlist.csv lookup with scanner source IPs, exclude in query
Marketing campaigns / sales events Spike correlates with known event (Black Friday, product launch, email blast) Temporarily adjust baseline during known events OR add scheduled suppression
New WAF rule deployment Spike starts exactly when new WAF rule was enabled, affects all traffic uniformly Monitor for 48h after any WAF rule change before trusting alerts; recalculate baseline
Partner API misconfiguration Single known partner IP, legitimate business traffic, not malicious request patterns Add partner IPs to known_partners lookup, potentially create separate "partner anomaly" detection
CDN health checks / edge probes Akamai/Cloudflare/Fastly source IP ranges, regular interval patterns Exclude CDN probe IP ranges using cidrmatch against published ranges
Load testing / performance testing Spike during QA testing windows, JMeter/Gatling/Locust user agents Coordinate with QA team for test schedules, suppress during known test windows
Bot traffic spikes Search engine crawlers, legitimate bots with proper identification Verify User-Agent, allowlist Googlebot/Bingbot verified IPs
