WAF-UC12: Rate Limit Trigger Analysis

Analyze patterns when rate limiting is triggered repeatedly. Persistent rate limit hits indicate sustained attack attempts, and patterns reveal attacker behavior and potential evasion techniques.

Severity HIGH
MITRE T1498
Schedule Every 5 min
Time Range Last 15 min

Why Analyze Rate Limit Triggers

Rate limits stop attacks, but the pattern tells the story:

• One-time trigger: Might be legitimate high traffic or minor misconfiguration

• Repeated triggers over hours: Persistent attacker testing limits

• Multiple IPs hitting limits: Distributed attack or botnet

• Specific endpoint triggers: Targeted abuse attempt

Step A: Build SPL in Search First

Where: Splunk Search → Build and test SPL

Variant A: Per-IP Rate Limit Analysis

Use when: You want to identify individual attackers hitting rate limits repeatedly. 

index=akamai sourcetype=akamai:siem earliest=-15m
| spath
| eval waf_action=coalesce(action, mvindex('attackData.rules{}.ruleAction',0))
| eval rule_name=coalesce(rule_name,'attackData.rules{}.ruleTag')
| eval src_ip=coalesce(src_ip,'attackData.clientIP')
| eval uri=coalesce(uri,'httpMessage.requestUri')
| eval vhost=coalesce(vhost,'httpMessage.requestHeaders.Host')
| where match(waf_action, "(?i)rate") OR match(rule_name, "(?i)rate")
| stats count as rate_triggers
        dc(uri) as unique_endpoints
        values(uri) as targeted_endpoints
        earliest(_time) as first_trigger
        latest(_time) as last_trigger
        BY src_ip, vhost
| where rate_triggers >= 5
| eval attack_duration_min = round((last_trigger - first_trigger) / 60, 1)
| eval triggers_per_min = round(rate_triggers / if(attack_duration_min > 0, attack_duration_min, 1), 1)
| eval attack_pattern = case(
    unique_endpoints < 3 AND rate_triggers > 20, "Focused Hammering - single target",
    unique_endpoints > 10 AND rate_triggers > 50, "Broad Scanning - many targets",
    rate_triggers > 100, "Sustained High Volume Attack",
    1=1, "Moderate Rate Abuse"
)
| eval is_auth_target = if(match(mvjoin(targeted_endpoints,","), "(?i)(login|auth|api|token)"), "Yes", "No")
| eval severity = case(
    rate_triggers > 100, "critical",
    is_auth_target="Yes" AND rate_triggers > 20, "critical",
    rate_triggers > 50, "high",
    is_auth_target="Yes", "high",
    1=1, "medium"
)
| table src_ip vhost rate_triggers unique_endpoints attack_duration_min triggers_per_min attack_pattern is_auth_target targeted_endpoints severity

Variant B: Aggregate Rate Limit Analysis (Distributed Detection)

Use when: You want to identify distributed attacks where multiple IPs are all hitting rate limits. 

index=akamai sourcetype=akamai:siem earliest=-15m
| spath
| eval waf_action=coalesce(action, mvindex('attackData.rules{}.ruleAction',0))
| eval rule_name=coalesce(rule_name,'attackData.rules{}.ruleTag')
| eval src_ip=coalesce(src_ip,'attackData.clientIP')
| eval uri=coalesce(uri,'httpMessage.requestUri')
| eval vhost=coalesce(vhost,'httpMessage.requestHeaders.Host')
| where match(waf_action, "(?i)rate") OR match(rule_name, "(?i)rate")
| stats count as total_triggers
        dc(src_ip) as unique_attackers
        dc(uri) as unique_endpoints
        values(src_ip) as sample_ips
        values(uri) as sample_endpoints
        BY vhost
| where total_triggers >= 20
| eval avg_per_ip = round(total_triggers / unique_attackers, 1)
| eval attack_type = case(
    unique_attackers > 50 AND avg_per_ip < 5, "Distributed Low-Rate Attack (botnet)",
    unique_attackers > 20, "Coordinated Multi-Source Attack",
    unique_attackers > 5, "Small Distributed Attack",
    1=1, "Single Source High Volume"
)
| eval severity = case(
    unique_attackers > 50 AND total_triggers > 100, "critical",
    unique_attackers > 20 AND total_triggers > 50, "high",
    total_triggers > 100, "high",
    1=1, "medium"
)
| table vhost total_triggers unique_attackers avg_per_ip unique_endpoints attack_type sample_ips sample_endpoints severity

Response Actions

Based on Attack Pattern

PatternSeverityRecommended Action
Focused Hammering HIGH Block IP immediately; investigate target endpoint for vulnerabilities
Auth/API Target CRITICAL Block IP; review auth logs for successful logins; check for credential stuffing
Distributed Attack CRITICAL Engage DDoS mitigation; consider temporary geo-blocking; contact Akamai SOC
Broad Scanning HIGH Block IP; cross-reference with scanner detection; may be reconnaissance
Sustained Volume HIGH Tighten rate limits; consider CAPTCHA challenge; block if purely malicious

Step C: Handle False Positives

Common False Positives & Solutions

FP SourceHow to IdentifySolution
Legitimate high-traffic user Known customer, regular business activity Increase rate limit for specific IPs/accounts; add to allowlist
Partner API integration Known partner IPs, expected high volume Create separate rate limit tier for partners
Mobile app burst Mobile app UA, sync patterns Adjust rate limits for app endpoints; consider token bucket
Marketing campaign spike Correlates with known marketing push Temporarily increase rate limits during campaigns
CDN cache miss storm Cache invalidation causes surge to origin Coordinate with CDN team; not a security issue
