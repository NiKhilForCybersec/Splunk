WAF-UC7: WAF Bypass Detection

WAF allows a request through, but the backend returns a 500 error. This may indicate a successful WAF evasion where malicious payload reached and crashed the application.

Severity HIGH
MITRE T1190
Schedule Every 5 min
Time Range Last 10 min

Why This Detection is Critical

⚠️ WAF Blocks ≠ Full Protection

Most detections focus on what the WAF blocked. But what if the attack got through?

This detection catches the scenario: WAF action = "allow" + Backend status = 500

If an attacker's payload bypassed the WAF and crashed your backend, you have a vulnerability that needs immediate patching.

Step 1: Decision Tree (Detection Logic)

flowchart TD
    START[WAF Event Received] --> A{Is waf_action = allow or pass or none}
    A -->|No blocked| OTHER[Already blocked - handled by other UCs]
    A -->|Yes allowed through| B{Is backend status 5xx}
    B -->|No| C{Is backend status 4xx with suspicious pattern}
    C -->|No| NORMAL[Normal allowed traffic - no issue]
    C -->|Yes| INVESTIGATE[Possible auth bypass or access control issue]
    B -->|Yes| D[Request allowed through but crashed backend]
    D --> E[Analyze the request that caused the error]
    E --> F{Does URI contain attack patterns}
    F -->|Yes| BYPASS[CONFIRMED: WAF Bypass - Attack reached backend]
    F -->|No| G{Is error rate from this IP elevated}
    G -->|Yes| SUSPICIOUS[Suspicious - possible fuzzing or testing]
    G -->|No| APPBUG[Likely application bug not attack]
    BYPASS --> H{Is attack pattern SQLi or RCE}
    H -->|Yes| CRIT[CRITICAL - Database or system compromise risk]
    H -->|No| HIGH1[HIGH - Attack payload reached application]
    SUSPICIOUS --> MED[MEDIUM - Needs investigation]
    INVESTIGATE --> MED
    CRIT --> CREATE[Create Notable Event]
    HIGH1 --> CREATE
    MED --> CREATE
    CREATE --> ENRICH[Include full request URI and error details]

Step A: Build SPL in Search First

Where: Splunk Search → Build and test SPL

Variant A: Basic WAF Bypass Detection

index=akamai sourcetype=akamai:siem earliest=-10m
| spath
| eval waf_action=coalesce(action, mvindex('attackData.rules{}.ruleAction',0), "allow")
| eval status=coalesce(status,'httpMessage.status')
| eval src_ip=coalesce(src_ip,'attackData.clientIP')
| eval uri=coalesce(uri,'httpMessage.requestUri')
| eval vhost=coalesce(vhost,'httpMessage.requestHeaders.Host')
| where (waf_action="allow" OR waf_action="pass" OR waf_action="none" OR isnull(waf_action))
| where status >= 500 AND status < 600
| stats count as error_count
        dc(uri) as unique_uris
        values(uri) as error_uris
        values(status) as error_codes
        BY src_ip, vhost
| where error_count >= 3
| eval has_attack_pattern = if(match(mvjoin(error_uris,","), "(?i)(select|union|script|\.\.\/|%00|cmd=|exec|eval)"), "Yes", "No")
| eval severity = case(
    has_attack_pattern="Yes" AND error_count > 10, "critical",
    has_attack_pattern="Yes", "high",
    error_count > 20, "high",
    1=1, "medium"
)
| eval bypass_type = case(
    has_attack_pattern="Yes", "CONFIRMED WAF BYPASS - Attack patterns in allowed requests",
    error_count > 20, "Possible fuzzing or automated testing",
    1=1, "Elevated 5xx errors - investigate"
)
| table src_ip vhost error_count unique_uris error_codes has_attack_pattern bypass_type error_uris severity

Variant B: With Attack Pattern Classification

index=akamai sourcetype=akamai:siem earliest=-10m
| spath
| eval waf_action=coalesce(action, mvindex('attackData.rules{}.ruleAction',0), "allow")
| eval status=coalesce(status,'httpMessage.status')
| eval src_ip=coalesce(src_ip,'attackData.clientIP')
| eval uri=coalesce(uri,'httpMessage.requestUri')
| eval vhost=coalesce(vhost,'httpMessage.requestHeaders.Host')
| where (waf_action="allow" OR waf_action="pass" OR isnull(waf_action))
| where status >= 500 AND status < 600
| eval attack_type = case(
    match(uri, "(?i)(select|union|insert|update|delete|drop|truncate)"), "SQLi",
    match(uri, "(?i)(script|onerror|onload|javascript)"), "XSS",
    match(uri, "(?i)(\.\.\/|\.\.\\\\|%2e%2e)"), "Path Traversal",
    match(uri, "(?i)(cmd=|exec=|system\(|eval\()"), "Command Injection",
    match(uri, "(?i)(%00|%0a|%0d)"), "Null Byte / CRLF",
    1=1, "Unknown"
)
| stats count as error_count
        dc(uri) as unique_uris
        values(uri) as sample_uris
        values(attack_type) as detected_attacks
        values(status) as error_codes
        BY src_ip, vhost
| where error_count >= 3
| eval severity = case(
    match(mvjoin(detected_attacks,","), "(SQLi|Command Injection)"), "critical",
    match(mvjoin(detected_attacks,","), "(XSS|Path Traversal)"), "high",
    error_count > 20, "high",
    1=1, "medium"
)
| table src_ip vhost error_count unique_uris detected_attacks error_codes sample_uris severity

Response Actions

If WAF Bypass Confirmed

  1. Block the source IP immediately at WAF or firewall level
  2. Review application logs for the time of the 500 errors
  3. Check for data exfiltration - did the attacker get a response before the error?
  4. Create new WAF rule to block the specific attack pattern
  5. Engage application team to patch the underlying vulnerability
  6. Preserve logs for forensic analysis

Step C: Handle False Positives

Common False Positives & Solutions

FP SourceHow to IdentifySolution
Application bugs 500 errors happen on normal URIs with no attack patterns Filter to only include URIs with attack pattern matches
Backend maintenance Spike in 500s correlates with deployment or maintenance window Correlate with change management; suppress during known windows
Database connection issues Widespread 500s across all URIs, not specific patterns Check for pattern diversity; single URI pattern = attack, all URIs = infra issue
Load testing High volume from known test IPs, expected patterns Allowlist load testing source IPs
