MFA-UC4: New Location Login Detection

First-time login from a country or city the user has never accessed from before. Requires a location baseline lookup to compare against historical patterns.

Severity HIGH
MITRE T1078
Schedule Every 15 min
Baseline 90 days

How This Differs from UC3 (Impossible Travel)

UC3: Impossible Travel

  • Compares two consecutive logins
  • Uses speed calculation between them
  • Detects active credential sharing
  • Doesn't need baseline lookup

UC4: New Location

  • Compares against 90-day baseline
  • Detects first-ever login from location
  • Catches initial compromise
  • Requires baseline lookup to be maintained

Prerequisite: Create User Location Baseline

Step 0: Generate Location Baseline (Run Once, Then Schedule Daily)

This detection REQUIRES a baseline lookup. Run this query to create it:

```Generate user location baseline from 90 days of logins
index=cyberark sourcetype=cyberark:identity:audit earliest=-90d
| where EventType IN ("Cloud.Core.Login", "Cloud.Core.MfaAuthentication") AND Result="Success"
| eval user = Username
| eval city = City
| eval country = Country
| where isnotnull(city) AND isnotnull(country)
| stats dc(city) as city_count 
        dc(country) as country_count
        values(city) as known_cities
        values(country) as known_countries
        count as total_logins
        earliest(_time) as first_seen
        latest(_time) as last_seen
        BY user
| eval days_active = round((last_seen - first_seen) / 86400, 0)
| table user total_logins city_count country_count known_cities known_countries days_active first_seen last_seen
| outputlookup user_location_baseline.csv
Schedule Daily Refresh
Save this as a scheduled report that runs daily (e.g., 3 AM) to keep baseline current. The baseline should capture travel patterns including occasional business travel.

Step A: Build SPL in Search First

Where: Splunk Search → Build and test SPL

New Location Detection with Baseline Comparison

index=cyberark sourcetype=cyberark:identity:audit earliest=-15m
| where EventType IN ("Cloud.Core.Login", "Cloud.Core.MfaAuthentication") AND Result="Success"
| eval user = Username
| eval city = City
| eval country = Country
| eval src_ip = FromIPAddress
| where isnotnull(city) AND isnotnull(country)
| lookup user_location_baseline.csv user OUTPUT known_cities known_countries total_logins
| where isnotnull(known_cities) AND isnotnull(known_countries)
| eval is_new_country = if(NOT match(known_countries, country), "Yes", "No")
| eval is_new_city = if(is_new_country="No" AND NOT match(known_cities, city), "Yes", "No")
| where is_new_country="Yes" OR is_new_city="Yes"
| eval high_risk_countries = "CN,RU,KP,IR,BY,NG"
| eval is_high_risk = if(match(high_risk_countries, country), "Yes", "No")
| lookup privileged_users.csv userPrincipalName AS user OUTPUT is_privileged role department
| eval location_type = case(
    is_new_country="Yes" AND is_high_risk="Yes", "NEW HIGH RISK COUNTRY",
    is_new_country="Yes", "NEW COUNTRY",
    is_new_city="Yes", "NEW CITY"
)
| eval severity = case(
    is_new_country="Yes" AND is_high_risk="Yes" AND is_privileged="true", "critical",
    is_new_country="Yes" AND is_high_risk="Yes", "high",
    is_new_country="Yes" AND is_privileged="true", "high",
    is_new_country="Yes", "medium",
    is_new_city="Yes" AND is_privileged="true", "medium",
    1=1, "low"
)
| table _time user role department is_privileged city country location_type is_high_risk known_cities known_countries total_logins src_ip severity

Step C: Handle False Positives

Common False Positives & Solutions

FP SourceHow to IdentifySolution
Business travel User is on legitimate business trip to new country Integrate with travel booking system if available; or quick user verification
New employee User has less than 30 days of login history Exclude users with less than 30 days of baseline; flag as "establishing baseline"
Vacation travel User logs in from vacation destination Lower severity for standard users; still investigate for privileged
VPN exit location VPN routes through different country than user's actual location Identify VPN IP ranges; tag these differently
Cloud proxy Zscaler/Netskope routes through regional data centers Identify cloud proxy IP ranges; these may change user's apparent location
Stale baseline User traveled months ago, baseline reset since then Keep baseline for 90+ days; don't overwrite with short windows
