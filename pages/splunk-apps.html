Splunk Apps & Add-ons Guide

Understanding Technology Add-ons (TAs), Apps, and how to install and manage them in Splunk Cloud

Apps vs Technology Add-ons (TAs)

Type Purpose Contains Examples
App User interface & workflows Dashboards, views, saved searches, navigation Splunk Enterprise Security, IT Service Intelligence
Technology Add-on (TA) Data parsing & normalization props.conf, transforms.conf, field extractions, CIM mappings Splunk_TA_Akamai, Splunk_TA_windows
Supporting Add-on (SA) Shared components for ES Lookups, macros, data models SA-ThreatIntelligence, SA-IdentityManagement
Domain Add-on (DA) Domain-specific content Correlation searches, dashboards for specific domains DA-ESS-AccessProtection
Key Insight
TAs handle the "how" — Apps handle the "what"
A TA tells Splunk how to parse Akamai logs. An App provides dashboards to visualize that data.

Finding Apps on Splunkbase

Splunkbase: splunkbase.splunk.com

The official marketplace for Splunk apps and add-ons.

Search Tips

  • Search by vendor name: "Akamai", "CyberArk", "Microsoft"

  • Check "Splunk Built" for official Splunk TAs
  • Sort by "Most Installs" for popular/trusted apps

Installing Apps in Splunk Cloud

Splunk Cloud Restrictions
Unlike on-premises Splunk, you cannot directly upload apps to Splunk Cloud. Apps must be:
• Cloud-vetted and available via self-service install, OR
• Installed via support case for private/custom apps

Method 1: Self-Service Install (Victoria Experience)

  1. Navigate to Apps → Find More Apps
  2. Search for the app/TA you need
  3. Click Install
  4. Accept any permissions prompts
  5. App installs automatically (may take a few minutes)

Note: Only cloud-vetted apps appear in self-service. If you don't see an app, use Method 2.

Method 2: Support Case (Private/Custom Apps)

  1. Download the app .tgz/.spl file from Splunkbase
  2. Open a support case at splunk.com/support
  3. Request app installation, attach the file
  4. Specify which stack (prod/dev) to install on
  5. Wait for confirmation (usually 1-2 business days)

Method 3: ACS API (Admin Config Service)

For programmatic installation (requires ACS access):

# Install app via ACS API curl -X POST "https://admin.splunk.com/<stack>/adminconfig/v2/apps" \ -H "Authorization: Bearer <token>" \ -H "Content-Type: application/json" \ -d '{"splunkbaseID": "1234", "version": "1.0.0"}'

Verifying App Installation

# List all installed apps | rest /services/apps/local | table title version disabled | sort title # Check specific app | rest /services/apps/local/Splunk_TA_Akamai | table title version disabled # Check if TA is parsing data index=akamai | head 1 | table sourcetype, _raw # Verify CIM fields are working index=akamai | head 10 | table src, dest, action, http_method

Common Installation Issues

Issue Symptom Solution
App not appearing Can't find in Apps menu Refresh browser, check disabled status via REST
Fields not extracting CIM fields are null Verify sourcetype matches TA expectations
Permission denied Can't access app features Check role permissions, app sharing settings
Version conflict App won't install Check Splunk version compatibility on Splunkbase

Configuring Apps After Installation

TA Configuration Pattern

Most TAs require these configuration steps:

  1. Create inputs — Configure how data arrives (HEC token, forwarder, etc.)
  2. Set sourcetype — Ensure data uses the TA's expected sourcetype
  3. Verify parsing — Check that fields are extracted correctly
  4. Confirm CIM mapping — Verify data model compliance

Example: Configuring Splunk_TA_Akamai

# 1. The TA expects sourcetype: akamai:siem # Configure your HEC token to use this sourcetype # 2. Verify data is being parsed index=akamai sourcetype=akamai:siem | head 5 # 3. Check field extractions index=akamai | fieldsummary | where count>0 # 4. Verify CIM compliance | tstats count from datamodel=Web where index=akamai by Web.src

Understanding App Structure

# Typical TA directory structure Splunk_TA_vendor/ ├── default/ │ ├── app.conf # App metadata │ ├── props.conf # Parsing configuration │ ├── transforms.conf # Field transforms │ ├── eventtypes.conf # Event categorization │ ├── tags.conf # CIM tagging │ └── data/ │ └── ui/ │ └── views/ # Any dashboards ├── local/ # Your customizations go here │ ├── props.conf # Override defaults │ └── transforms.conf ├── lookups/ # Lookup tables ├── metadata/ │ └── default.meta # Permission settings └── README.txt
Never Edit default/ Files
Always put customizations in local/ directory. Changes to default/ will be overwritten on app updates.

Enterprise Security Content Update (ESCU)

What is ESCU?

ESCU (also called "ES Content Update" or "Splunk Security Essentials") provides:

  • 1,500+ pre-built detections organized by threat type
  • Analytic Stories — Grouped detections for specific threats
  • MITRE ATT&CK mapping — Every detection mapped to framework
  • Regular updates — New detections added frequently

Installing ESCU

# ESCU is typically pre-installed with ES # To verify: | rest /services/apps/local/DA-ESS-ContentUpdate | table title version # List all Analytic Stories | rest /services/saved/searches | search "action.escu.analytic_story"="*" | stats dc(title) as detection_count by action.escu.analytic_story

Key Analytic Stories for WAF/Web Security

Analytic Story Detections Relevant Data
Apache Struts Vulnerability Exploit attempts WAF, Web logs
SQL Injection SQLi patterns in requests WAF, Web logs
Suspicious Web Traffic Anomalous user agents, paths WAF, Proxy
Web Fraud Detection Account enumeration, credential stuffing WAF, Auth logs

Discovering ESCU Detections via SPL

# Find all detections for "Web" data model | rest /services/saved/searches | search title="*" action.correlationsearch.enabled=1 | search search="*datamodel=Web*" OR search="*Web.*" | table title, search, action.correlationsearch.annotations # Find detections by MITRE technique | rest /services/saved/searches | where match('action.correlationsearch.annotations', "T1190") | table title, search

Quick Reference Commands

# List all apps with versions | rest /services/apps/local | table title version disabled | sort title # Check app permissions | rest /services/apps/local | table title sharing perms.read perms.write # Find which app a saved search belongs to | rest /services/saved/searches | table title eai:acl.app # Check for disabled apps | rest /services/apps/local | where disabled=1 | table title # App configuration via REST | rest /services/configs/conf-props | search title="akamai*" | table title, stanza
