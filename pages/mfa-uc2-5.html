MFA Use Cases (UC2-UC5)

Complete MFA detection suite for privileged account protection

CyberArk is PRIMARY for Privileged MFA
All MFA detections for privileged users should query CyberArk logs FIRST. Entra ID is secondary for standard corporate users.

MFA-UC2: MFA Failure Burst ‚Üí Success

Severity
HIGH
MITRE ATT&CK
T1110, T1621
Primary Source
CyberArk
Secondary
Entra ID

MFA Failure Burst ‚Üí Success detects when multiple MFA failures are immediately followed by a successful authentication, indicating the attacker finally succeeded in bypassing MFA.

sequenceDiagram participant A as Attacker participant I as CyberArk/Entra participant V as Victim A->>I: Login + MFA I->>V: Push notification V-->>I: ‚ùå Deny A->>I: Login + MFA (retry) I->>V: Push notification V-->>I: ‚ùå Deny A->>I: Login + MFA (retry) I->>V: Push notification V-->>I: ‚ùå Deny Note over A,V: 3+ failures in short time A->>I: Login + MFA (persistence) I->>V: Push notification V-->>I: ‚úÖ Approve (fatigue/mistake) I->>A: ‚úÖ Access Granted Note over A: üî¥ ACCOUNT COMPROMISED
KEY DIFFERENCE from UC1
UC1 detects fatigue attacks in progress. UC2 detects SUCCESSFUL compromise ‚Äî failures followed by success means the attacker got in.

A legitimate user typically succeeds on their first MFA attempt. Multiple failures followed by success is abnormal and indicates:

  • User gave in to MFA fatigue attack
  • Attacker socially engineered user to approve
  • SIM swap completed and SMS OTP intercepted
  • Device compromise allowing MFA bypass
MFA-UC2: Failure Burst ‚Üí Success (CyberArk) 
index=cyberark sourcetype=cyberark:identity:audit earliest=-15m
    EventType IN ("Cloud.Core.MfaDenied", "Cloud.Core.MfaTimeout", "Cloud.Core.MfaAuthentication")
| spath
| eval user = ActorId
| eval mfa_failed = if(Result IN ("Failure", "Denied", "Timeout"), 1, 0)
| eval mfa_success = if(Result=="Success", 1, 0)

``` Filter to privileged users
| lookup privileged_users.csv userPrincipalName AS user OUTPUT is_privileged role

``` Aggregate by user in time window
| bin _time span=10m
| stats 
    sum(mfa_failed) as failures
    sum(mfa_success) as successes
    min(_time) as first_event
    max(_time) as last_event
    values(ClientIP) as source_ips
    latest(role) as role
    BY user _time

``` KEY CONDITION: Failures followed by success
| where failures >= 3 AND successes >= 1

``` This is a COMPROMISE indicator
| eval alert_severity = "CRITICAL"
| eval alert_message = "MFA bypass detected - ".failures." failures followed by success"

| table user role failures successes source_ips first_event last_event alert_severity alert_message

MFA-UC3: Privileged Access from New Location/Device

Severity
MEDIUM
MITRE ATT&CK
T1078
Primary Source
CyberArk
Secondary
Entra ID

New Location/Device Detection alerts when a privileged user successfully authenticates from a location or device they've never used before.

flowchart TD A[Login Event] --> B{User Type?} B -->|Standard| C[Lower Priority] B -->|Privileged| D{Location Known?} D -->|In Baseline| E[Normal - No Alert] D -->|New Country| F[üü° ALERT: New Location] D -->|New City| G{Same Country?} G -->|Yes, Canada| H[Log for Review] G -->|No| F F --> I{Device Known?} I -->|New Device Too| J[üî¥ HIGH ALERT] I -->|Known Device| K[Medium Alert] style F fill:#d97706,color:#fff style J fill:#dc2626,color:#fff

Privileged users (IT admins, DBAs) typically work from predictable locations. A login from an unexpected location may indicate:

  • Credential theft being used from attacker's location
  • VPN/proxy being used to mask origin (suspicious)
  • Legitimate travel (verify with user)
Requires Baseline
Build a lookup of normal locations per user from 30+ days of historical data before enabling this detection.
MFA-UC3: New Location Detection (Entra ID) 
index=azure sourcetype=azure:aad:signin earliest=-1h
| spath
| eval user = 'properties.userPrincipalName'
| eval error_code = 'properties.status.errorCode'
| eval city = 'properties.location.city'
| eval country = 'properties.location.countryOrRegion'
| eval device_id = 'properties.deviceDetail.deviceId'

``` Only successful logins
| where error_code==0

``` Filter to privileged users
| lookup privileged_users.csv userPrincipalName AS user OUTPUT is_privileged role
| where is_privileged="true"

``` Compare against baseline locations
| lookup user_baseline_locations.csv user OUTPUT known_countries known_cities

``` Detect new locations
| eval new_country = if(isnull(mvfind(known_countries, country)), 1, 0)
| eval new_city = if(isnull(mvfind(known_cities, city)), 1, 0)

``` Alert on new location
| where new_country=1 OR new_city=1

| eval alert_type = case(
    new_country=1, "New Country - HIGH",
    new_city=1, "New City - MEDIUM"
)

| table _time user role city country new_country new_city device_id alert_type

MFA-UC4: Impossible Travel Detection

Severity
HIGH
MITRE ATT&CK
T1078
Logic
Geo + Time
Speed Limit
>800 km/h

Impossible Travel detects when a user logs in from two geographically distant locations in a time frame that would be physically impossible to travel.

flowchart LR subgraph Login1[10:00 AM] A[Toronto, Canada] end subgraph Login2[10:30 AM] B[Moscow, Russia] end A -->|"7,500 km in 30 min"| B B --> C{Speed Check} C -->|"15,000 km/h > 800 km/h"| D[üî¥ IMPOSSIBLE TRAVEL] style D fill:#dc2626,color:#fff
Travel Speed Possible? Action
<200 km/h ‚úÖ Driving/Train No alert
200-800 km/h ‚úÖ Commercial flight No alert
>800 km/h ‚ùå Impossible ALERT
MFA-UC4: Impossible Travel (Entra ID with Haversine) 
index=azure sourcetype=azure:aad:signin earliest=-24h
| spath
| eval user = 'properties.userPrincipalName'
| eval error_code = 'properties.status.errorCode'
| eval lat = 'properties.location.geoCoordinates.latitude'
| eval lon = 'properties.location.geoCoordinates.longitude'
| eval city = 'properties.location.city'
| eval country = 'properties.location.countryOrRegion'

``` Only successful logins with geo data
| where error_code==0 AND isnotnull(lat) AND isnotnull(lon)

``` Get previous login for each user
| sort user _time
| streamstats current=f 
    last(lat) as prev_lat 
    last(lon) as prev_lon
    last(_time) as prev_time 
    last(city) as prev_city
    last(country) as prev_country
    BY user

``` Skip first login (no previous)
| where isnotnull(prev_lat)

``` Haversine formula for distance in km
| eval lat1_rad = lat * 3.14159265359 / 180
| eval lat2_rad = prev_lat * 3.14159265359 / 180
| eval dlon_rad = (lon - prev_lon) * 3.14159265359 / 180
| eval dlat_rad = (lat - prev_lat) * 3.14159265359 / 180
| eval a = pow(sin(dlat_rad/2), 2) + cos(lat1_rad) * cos(lat2_rad) * pow(sin(dlon_rad/2), 2)
| eval distance_km = 6371 * 2 * asin(sqrt(a))

``` Calculate speed
| eval time_diff_hours = (_time - prev_time) / 3600
| eval speed_kmh = if(time_diff_hours > 0, distance_km / time_diff_hours, 0)

``` Flag impossible travel
| where speed_kmh > 800 AND distance_km > 500

``` Lookup privileged status
| lookup privileged_users.csv userPrincipalName AS user OUTPUT is_privileged role
| eval severity = if(is_privileged="true", "CRITICAL", "HIGH")

``` Format output
| eval distance_km = round(distance_km, 0)
| eval speed_kmh = round(speed_kmh, 0)
| eval time_diff_min = round(time_diff_hours * 60, 0)

| table user role prev_city prev_country city country distance_km time_diff_min speed_kmh severity

MFA-UC5: MFA Method Downgrade

Severity
HIGH
MITRE ATT&CK
T1556
Focus
Method Change
Risk
Bypass Attempt

MFA Method Downgrade detects when a user switches from a stronger MFA method to a weaker one, which may indicate an attacker trying to use a more vulnerable factor.

MFA Method Security Ranking

Rank Method Security Vulnerability
1 (Best) FIDO2 / Hardware Key Highest Physical theft only
2 Authenticator (Number Match) High Resistant to fatigue
3 Authenticator (Push) Medium MFA fatigue vulnerable
4 TOTP App Medium Phishing possible
5 (Worst) SMS / Voice Low SIM swap, SS7 attacks
Example Attack Scenario
Attacker has stolen password + completed SIM swap ‚Üí Changes MFA from Authenticator to SMS ‚Üí Now receives OTP codes.
MFA-UC5: Method Downgrade Detection 
index=azure sourcetype=azure:aad:signin earliest=-24h
| spath
| eval user = 'properties.userPrincipalName'
| eval mfa_method = 'properties.mfaDetail.authMethod'
| eval error_code = 'properties.status.errorCode'

``` Only successful MFA authentications
| where error_code==0 AND isnotnull(mfa_method)

``` Assign security score to methods
| eval method_score = case(
    match(mfa_method, "(?i)fido"), 5,
    match(mfa_method, "(?i)windows.?hello"), 5,
    match(mfa_method, "(?i)number.?match"), 4,
    match(mfa_method, "(?i)authenticator"), 3,
    match(mfa_method, "(?i)totp"), 3,
    match(mfa_method, "(?i)sms|text"), 1,
    match(mfa_method, "(?i)voice|phone|call"), 1,
    1=1, 2
)

``` Get previous method for user
| sort user _time
| streamstats current=f last(mfa_method) as prev_method last(method_score) as prev_score BY user

``` Detect downgrade
| where isnotnull(prev_score) AND method_score < prev_score

``` Calculate downgrade severity
| eval downgrade_severity = prev_score - method_score
| eval alert_level = case(
    downgrade_severity >= 3, "CRITICAL",
    downgrade_severity >= 2, "HIGH",
    1=1, "MEDIUM"
)

``` Filter to privileged for highest priority
| lookup privileged_users.csv userPrincipalName AS user OUTPUT is_privileged role

| table _time user role prev_method mfa_method prev_score method_score downgrade_severity alert_level is_privileged
| sort - downgrade_severity
