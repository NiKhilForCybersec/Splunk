Complete guide: TA Installation ‚Üí Data Configuration ‚Üí Validation ‚Üí ESCU Detections ‚Üí Tuning

üèóÔ∏è Understanding Detection Sources in Splunk ES

Splunk ES has two sources of OOTB detections:

  1. ES Native Correlation Searches ‚Äî Built into Enterprise Security (e.g., "Brute Force Access Behavior Detected")
  2. ESCU (Enterprise Security Content Update) ‚Äî 1,900+ detections maintained by Splunk Threat Research Team, organized into "Analytic Stories"

Key insight: There is no published list of all OOTB searches ‚Äî you must discover them via SPL queries in your environment.

Install Splunk Add-on for Akamai (Splunk Cloud)

Self-service or support case depending on your Cloud tier

‚òÅÔ∏è Splunk Cloud Note: App installation in Splunk Cloud varies by your deployment type (Victoria Experience vs Classic Experience). Check via: Support & Services ‚Üí About
Option A: Self-Service via Splunk Web (Most Deployments)

Available for most Splunk Cloud tiers with sc_admin role:

  1. In Splunk Web, click Apps ‚Üí Find More Apps
  2. Search for Splunk Add-on for Akamai
  3. Verify publisher is Splunk Inc. (App ID: Splunk_TA_Akamai)
  4. Click Install
  5. Enter your Splunk.com credentials
  6. Accept license terms and install
Option B: Admin Config Service (ACS) API

For programmatic/CI-CD installation:

Install via ACS API
``` 1. Get your Splunkbase session ID (from browser dev tools after login)
``` 2. Get JWT token from Splunk Cloud (Settings ‚Üí Tokens)
``` 3. Install via curl:

curl -X POST 'https://admin.splunk.com/{stack}/adminconfig/v2/apps/victoria?splunkbase=true' \
  --header 'X-Splunkbase-Authorization: {splunkbase_session_id}' \
  --header 'Authorization: Bearer {jwt_token}' \
  --header 'ACS-Licensing-Ack: {license_url}' \
  --header 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'splunkbaseID=1552' \
  --data-urlencode 'version=2.0.0'
Option C: Support Case (Restricted Apps)

If self-service unavailable:

  1. Open a Splunk Support Case via support portal
  2. Subject: "App Installation Request - Splunk Add-on for Akamai"
  3. Include: Stack name, App ID (Splunk_TA_Akamai), Splunkbase link, business justification
  4. Typical turnaround: 1-2 business days

Validate Data Flow & Field Extraction

Confirm data arrives and TA parses correctly

Validation Queries ‚Äî Run Each One
``` ========== TEST 1: Data Arriving? ==========
index=akamai earliest=-1h | stats count
| eval status=if(count>0, "‚úÖ PASS", "‚ùå FAIL - Check HEC config")
| table count status

``` ========== TEST 2: Correct Sourcetype? ==========
index=akamai earliest=-1h | stats count by sourcetype
``` Expected: akamai:siem
``` If you see "httpevent" ‚Üí Wrong sourcetype in HEC token config

``` ========== TEST 3: Fields Extracting? (TA Working?) ==========
index=akamai earliest=-1h | head 10
| table _time, src_ip, action, http_uri, http_method, rule_id, http_user_agent
``` All columns should have values
``` Empty fields ‚Üí TA not installed or wrong sourcetype

``` ========== TEST 4: Data Freshness / Latency ==========
index=akamai earliest=-1h 
| stats max(_time) as latest_event 
| eval lag_minutes=round((now()-latest_event)/60,1)
| eval status=case(lag_minutes<5,"‚úÖ Excellent",lag_minutes<15,"‚úÖ Good",lag_minutes<60,"‚ö†Ô∏è Acceptable",true(),"‚ùå Check Akamai config")
| table latest_event, lag_minutes, status

``` ========== TEST 5: CIM Normalization (Web Data Model) ==========
| tstats count from datamodel=Web by sourcetype, index | head 10
``` If akamai data appears ‚Üí CIM mapping working
``` If empty ‚Üí Data model not accelerated or CIM Add-on missing

``` ========== TEST 6: Volume Check (24h) ==========
index=akamai earliest=-24h | timechart span=1h count
``` Look for consistent pattern - gaps indicate connectivity issues

Discover Available Detections

Find ES native and ESCU detections relevant to WAF data

‚ö†Ô∏è Important: No Published List Exists

Splunk does not publish a complete list of OOTB correlation searches. You must query your environment to discover available detections. The searches below help you find what's available.

Step 4.1: Find ES Native Correlation Searches
List all ES correlation searches (excluding ESCU)
| rest splunk_server=local count=0 /services/saved/searches 
| where match('action.correlationsearch.enabled', "1|[Tt]|[Tt][Rr][Uu][Ee]") 
| where NOT match(title, "^ESCU") 
| rename title as "Detection Name", 
         action.notable.param.security_domain as "Domain",
         disabled as "Disabled"
| table "Detection Name", "Domain", "Disabled", description
| sort "Detection Name"

Common ES native searches for web/network: "Brute Force Access Behavior Detected", "Concurrent Login Attempts Detected", "High Volume of Traffic from Single Source"

Step 4.2: Find ESCU Detections (Web/Network Category)
List ESCU detections relevant to web attacks
| rest splunk_server=local count=0 /services/saved/searches 
| search title="ESCU*" OR eai:acl.app="*ESCU*" OR eai:acl.app="DA-ESS-ContentUpdate"
| where match('action.correlationsearch.enabled', "1|[Tt]|[Tt][Rr][Uu][Ee]")
| search title="*Web*" OR title="*HTTP*" OR title="*Request*" OR title="*Attack*" 
         OR title="*Smuggling*" OR title="*Injection*" OR title="*XSS*"
| rename title as "Detection Name", disabled as "Disabled"
| table "Detection Name", "Disabled", description
| sort "Detection Name"
Step 4.3: Explore ESCU Analytic Stories (Use Case Library)

Navigate to: ES ‚Üí Security Content ‚Üí Security Use Case Library

Filter by relevant categories:

  • ‚úÖ HTTP Request Smuggling ‚Äî CL.TE, TE.TE, header manipulation attacks
  • ‚úÖ Suspicious User Agents ‚Äî Malware, PUA, RMM tool user agents
  • ‚úÖ Web Attack Detection ‚Äî SQLi, XSS, path traversal patterns
  • ‚úÖ Command and Control ‚Äî C2 beaconing, suspicious network patterns
  • ‚úÖ Data Exfiltration ‚Äî Large uploads, unusual destinations

Enable, Configure & Test Detections

Proper workflow for enabling and validating detections

Step 5.1: Enable a Detection
  1. Navigate to ES ‚Üí Configure ‚Üí Content ‚Üí Content Management
  2. Find the detection (use search/filter)
  3. Click detection name to open details
  4. Click Edit
  5. Change Status from "Disabled" to "Enabled"
  6. IMPORTANT: Review and modify the search (see Step 5.2)
  7. Configure scheduling (cron or interval)
  8. Click Save
Step 5.2: Modify Search for Your Data

Most OOTB searches use generic index patterns. You must customize:

Change FromChange ToWhy
index=*index=akamaiPerformance & accuracy
`web_index` macroVerify macro definitionMay need updating
Generic sourcetypesourcetype=akamai:siemFilter to your data
Step 5.3: Test Detection BEFORE Enabling
Test search against historical data
``` 1. Copy the detection's SPL search
``` 2. Paste into Search & Reporting
``` 3. Modify time range to last 7 days
``` 4. Run the search
``` 5. Check:
```    - Does it return results? (Validate logic works)
```    - Are results true positives? (Check a sample)
```    - Is volume acceptable? (Not thousands of results)

``` Example test for HTTP suspicious user agent:
index=akamai sourcetype=akamai:siem earliest=-7d
| search http_user_agent IN ("curl*", "wget*", "python*", "Go-http-client*")
| stats count by src_ip, http_user_agent, http_uri
| where count > 10
Step 5.4: Configure Throttling

Prevent alert fatigue by throttling repeated alerts:

Location:Detection Edit ‚Üí Throttling section
Throttle by field:src_ip or src (group alerts by attacker IP)
Window duration:30-60 minutes (typical)
Max notable events:1 per window (recommended start)
Step 5.5: Validate Detection is Firing
Check if detection created notable events
``` Check notable events from a specific detection:
index=notable search_name="*YourDetectionName*" earliest=-24h
| stats count by search_name, urgency, status_label
| sort -count

``` Check all recent notable events:
index=notable earliest=-24h
| stats count by search_name
| sort -count

``` Check detection run history:
| rest /services/saved/searches 
| search title="*YourDetectionName*"
| table title, next_scheduled_time, last_scheduled_time, 
        action.correlationsearch.enabled, disabled

Tune Detections & Create Custom Rules

Reduce false positives and fill detection gaps

Tuning Strategies
IssueSolutionExample
Too many alerts (FPs) Add exclusions / allowlist | search NOT src_ip IN ("10.0.0.*", "192.168.*")
Known scanners triggering Exclude known scanner IPs via lookup | lookup allowed_scanners.csv src_ip OUTPUT allowed | where isnull(allowed)
Threshold too sensitive Increase threshold based on baseline Change count > 10 to count > 100
Missing context Enrich with asset/identity lookups | lookup asset_lookup.csv dest_ip OUTPUT asset_criticality
When to Create Custom Detections
‚úÖ Use OOTB/ESCU For:
  • Generic attack patterns (SQLi, XSS)
  • Known malware user agents
  • Standard brute force patterns
  • Common C2 indicators
üîß Create Custom For:
  • Client-specific endpoints
  • Baseline anomaly detection
  • Business logic abuse
  • Geographic anomalies

‚Üí See WAF Use Cases (UC1-12) for custom detection templates
