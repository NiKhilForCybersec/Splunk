üìÅ How to Create an Index
What is an Index?

An index is where Splunk stores your data. Think of it like a folder or database. Each data source should have its own index (e.g., akamai, cyberark, azure) for organization, access control, and retention management.

Option A: Create It Yourself (If You Have Permissions)
  1. Go to Settings (top menu bar)
  2. Click Indexes
  3. Click "New Index" button (top right)
  4. Fill in:
    • Index Name: akamai (lowercase, no spaces)
    • Index Data Type: Events
    • Max Size: Ask Splunk admin for guidance (e.g., 100GB)
    • Searchable retention: e.g., 90 days
  5. Click Save

‚ö†Ô∏è If "New Index" is grayed out or you get permission error ‚Üí Use Option B

Option B: Request from Splunk Admin (Splunk Cloud)

In Splunk Cloud, index creation is often restricted. You need to request it:

Who to Contact:
  • Internal Splunk admin team at Sobeys
  • OR submit Splunk Cloud support case (if no internal admin)
Information to Include in Request:
Subject: Request: New Index Creation - Akamai WAF Logs

Index Name: akamai
Purpose: Store Akamai WAF security events for detection engineering
Data Source: Akamai SIEM Integration API
Expected Daily Volume: ~5-10 GB/day (estimate)
Retention Required: 90 days searchable
Sourcetype: akamai:siem
Project: WAF/DDoS Detection Engineering
Requested By: [Your Name]
Business Justification: Required for security monitoring and threat detection
Option C: Splunk Cloud Support Case

If there's no internal Splunk admin:

  1. Go to splunk.com/support
  2. Log in with your Splunk Cloud credentials
  3. Create new case ‚Üí "Cloud Operations"
  4. Use the information template above
  5. Typical turnaround: 1-3 business days

Akamai Validation Queries

Akamai Data Validation
``` 1. Check if data is flowing
index=akamai earliest=-1h | stats count
| eval status = if(count > 0, "‚úÖ Data flowing", "‚ùå No data")

``` 2. Check sourcetype
index=akamai earliest=-1h | stats count by sourcetype
``` Expected: akamai:siem

``` 3. Check data freshness (latency)
index=akamai earliest=-1h 
| stats max(_time) as latest 
| eval lag_min = round((now() - latest) / 60, 1)
| eval status = if(lag_min < 15, "‚úÖ OK", "‚ö†Ô∏è Delayed")

``` 4. Check field extraction
index=akamai earliest=-1h | head 10
| table _time src_ip dest_ip action http_method http_uri rule_id
``` All fields should have values

``` 5. Check event volume (baseline)
index=akamai earliest=-24h 
| timechart span=1h count
``` Look for consistent pattern, no gaps

``` 6. Check action distribution
index=akamai earliest=-24h 
| stats count by action
``` Expected: alert, deny, allow

Entra ID Special Considerations
Nested JSON: Entra logs use deeply nested JSON. Use spath or configure props.conf for automatic extraction.

Event Hub Latency: Expect 5-15 minute latency vs. near real-time for HEC sources. This is normal for Event Hub architecture.

Azure AD vs Entra ID: Same thing - Microsoft rebranded in 2023. Sourcetypes still use azure:aad:* naming.
