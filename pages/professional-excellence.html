Deep Knowledge: Understand the WHY

Don't just know WHAT to do ‚Äî understand WHY so you can explain and adapt.

Detection Engineering Fundamentals

ConceptWhat It IsWhy It MattersIf Asked, Explain It Like This
False Positive (FP) Alert fires but it's not actually malicious Too many FPs = SOC ignores alerts = real attacks missed "A false positive is when the detection alerts but it's legitimate activity. High FP rates cause alert fatigue ‚Äî the SOC starts ignoring alerts, which is worse than having no detection at all."
False Negative (FN) Attack happens but detection doesn't fire Missed attacks = breaches "A false negative is a missed attack ‚Äî the malicious activity happened but we didn't detect it. This is the worst outcome because we think we're protected but we're not."
Threshold The boundary that triggers an alert Too low = FPs, too high = FNs "The threshold is where we draw the line between normal and suspicious. I calculate it using statistical methods ‚Äî typically mean plus 2 standard deviations ‚Äî so it's based on YOUR data, not arbitrary guessing."
Baseline Normal behavior patterns in your environment Can't detect anomalies without knowing normal "A baseline is what 'normal' looks like in your environment. Without a baseline, we're just guessing. I collect 14 days minimum to capture weekday/weekend patterns before setting thresholds."
Silent Mode Detection runs but doesn't alert Test FP rate before bothering SOC "Silent mode means the detection is running and logging results, but not sending alerts to the SOC. This lets me measure the false positive rate in production conditions before we commit to live alerting."
CIM (Common Information Model) Splunk's standardized field naming Write once, run on any data source "CIM is Splunk's standardized field naming convention. If I write a detection using CIM fields like 'src_ip' instead of vendor-specific fields, the same detection works across multiple data sources without modification."
MITRE ATT&CK Industry framework of attack techniques Proves detection coverage, audit-friendly "MITRE ATT&CK is the industry-standard framework for categorizing attacker techniques. Mapping our detections to ATT&CK shows exactly what threats we cover and identifies gaps ‚Äî it's what auditors expect to see."

Why We Do Things This Way

PracticeWhy We Do ItWhat Happens If We Don't
14-day baseline before thresholds Captures weekday vs weekend, beginning vs end of month patterns Thresholds based on partial data = FPs on days we didn't measure
7-day silent mode before production Tests real-world FP rate before bothering SOC Deploy noisy detection = SOC loses trust = they ignore future alerts
Document every exclusion with justification Future tuner understands why; auditor sees it's intentional Undocumented exclusions look like coverage gaps or mistakes
Runbook for every detection SOC knows exactly what to do; consistent response Different analysts handle same alert differently = inconsistent security
Use statistical thresholds (mean + 2œÉ) Based on data, not opinion; defensible to auditors "Why 100?" "Because I thought so" ‚Äî not professional
Test against historical attacks Proves detection actually catches what it's supposed to Detection might look good but miss real attacks

Hard Questions & How to Answer

Be prepared for these ‚Äî especially from a skeptical manager.

Hard QuestionExpert Answer
"How do you know this detection actually works?" "I validate detections three ways: (1) Test against known attack patterns in historical data, (2) Run in silent mode for 7 days to measure real-world FP rate, (3) Have SOC review sample alerts to confirm they're actionable. The current FP rate is X%, which is within acceptable range."
"Why is this taking so long?" "Detection engineering has required phases: data validation (ensuring quality foundation), baseline collection (14 days to capture patterns), development, testing, and tuning. Skipping phases leads to unreliable detections that burden the SOC. I'm currently in [phase], on track for [date]."
"Can't we just use the vendor's built-in rules?" "Vendor rules are generic starting points. They don't know your environment ‚Äî your IP ranges, your service accounts, your business hours. I customize detections using your baseline data to reduce false positives and increase relevance. Generic rules typically have 30%+ FP rates; my target is under 10%."
"What if an attack happens and your detection misses it?" "No detection is 100%. My approach is defense in depth ‚Äî multiple detections covering different attack stages. I also document exactly what each detection covers using MITRE ATT&CK mapping, so we know our coverage and gaps. If something is missed, I do root cause analysis to determine if it's a gap we should address."
"How did you come up with that threshold?" "I calculated it from your 14-day baseline. The mean is X, standard deviation is Y, so threshold is mean + 2œÉ = Z. This means we alert on the top 5% of outliers statistically. I can show you the analysis if you'd like."
"Why should we trust your approach?" "My approach follows industry standards: MITRE ATT&CK for threat coverage, statistical methods for thresholds, Splunk best practices for implementation. Everything is documented and defensible. I'm happy to walk you through any aspect in detail."
