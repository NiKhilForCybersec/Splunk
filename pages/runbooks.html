RB-WAF-01: WAF Block Volume Anomaly

Alert Information

  • Severity: Medium (dynamic)
  • MITRE ATT&CK: T1190
  • Data Source: Akamai SIEM
  • SLA: Acknowledge 4h, Resolve 24h

Escalation Path

  • Tier 1: Initial triage, FP check
  • Tier 2: Deep analysis, rule review
  • Escalate if: Confirmed campaign
  1. Verify alert is not a known false positive
  2. Check block count and deviation â€” is this significant?
  3. Review unique_sources â€” single vs distributed attack?
  4. Check attack_types â€” what rules are triggering?
Pivot Query
index=akamai sourcetype=akamai:siem earliest=-15m
| spath path=attackData.rules{}.ruleAction output=action
| where action="deny"
| spath path=attackData.clientIP output=src
| spath path=attackData.rules{}.ruleTag output=category
| stats count BY src category
| sort - count
  1. Identify top source IPs contributing to spike
  2. Check IP reputation (AbuseIPDB, VirusTotal)
  3. Determine if sources are known (internal, partner, scanner)
  4. Check geographic distribution
Top Sources Query
index=akamai sourcetype=akamai:siem earliest=-15m
| spath path=attackData.rules{}.ruleAction output=action
| where action="deny"
| spath path=attackData.clientIP output=src
| spath path=geo.country output=country
| spath path=geo.asn output=asn
| stats count as blocks BY src country asn
| sort - blocks | head 20
True Positive Actions
  • Document source IPs, attack types, targets
  • Consider temporary IP block in Akamai
  • Notify application owners if targeted
  • Escalate to IR if coordinated campaign
  • Create incident ticket
False Positive Actions
  • Document FP reason
  • Add to allowlist if recurring
  • Adjust threshold if systematic
  • Close alert with FP disposition

RB-WAF-02: Single Source Flood

Alert Information

  • Severity: High
  • MITRE ATT&CK: T1498, T1499
  • Data Source: Akamai SIEM
  • SLA: Acknowledge 30m, Resolve 4h

Threshold Criteria

  • >500 requests from single IP, OR
  • >100 blocks from single IP, OR
  • >50% block ratio from single IP
High Volume = Potential DDoS
If single source is generating >1000 req/min, this may be an active DDoS attempt. Consider immediate block.
  1. Check total request volume from source IP
  2. Calculate requests per minute rate
  3. Review attack categories being triggered
  4. Check if application is experiencing degradation
  1. Check AbuseIPDB for IP reputation
  2. Check VirusTotal for known malicious activity
  3. Verify ASN ownership (hosting provider, residential, enterprise)
  4. Check if IP is in known VPN/proxy/Tor range
Source Type Risk Level Recommended Action
Known hosting provider High Likely malicious â€” consider blocking
VPN/Proxy service Medium Could be attacker hiding â€” monitor closely
Residential ISP Medium May be compromised device â€” rate limit
Enterprise network Low Could be misconfigured scanner â€” verify

Immediate (if >1000 req/min)

Request Akamai admin to temporarily block IP. Document ticket number.

Short-term (24-48 hours)

Implement IP-level rate limiting in Akamai policy. Monitor for evasion (new IPs).

Long-term

Add IP/range to permanent blocklist if confirmed malicious. Update threat intel feeds.

Never Auto-Close MFA Alerts
Always verify with user via phone before closing as FP. An attacker could claim "phone issues" if contacted via potentially compromised channels.

Alert Disposition Codes

True Positive Dispositions

  • TP-BLOCKED: Attack detected and blocked
  • TP-CONTAINED: Attack detected, containment actions taken
  • TP-ESCALATED: Escalated to Incident Response
  • TP-COMPROMISED: Confirmed compromise

False Positive Dispositions

  • FP-SCANNER: Legitimate security scanner
  • FP-PARTNER: Known partner/vendor traffic
  • FP-THRESHOLD: Threshold too sensitive
  • FP-BENIGN: Confirmed benign activity
