Lookup Types in Splunk

Type Storage Max Size Use Case Splunk Cloud
CSV Lookup File on search head ~100MB recommended Static reference data (users, assets) ✓ Supported
KV Store MongoDB backend ~1GB per collection Dynamic data, frequent updates ✓ Supported
External Lookup Python/script N/A (API calls) Real-time enrichment (threat intel) Limited
Geolocation MaxMind DB Built-in IP to location mapping ✓ Built-in
Asset/Identity ES Framework Varies ES asset correlation ES Required

Adding Lookups in Splunk Cloud

Splunk Cloud Consideration
In Splunk Cloud, you cannot directly access the filesystem. All lookups must be uploaded via the UI or API. For transforms.conf changes, you may need to submit a support ticket or use a custom app.

Method 1: Splunk Web UI (Recommended)

Step 1: Navigate to Lookups

Settings → Lookups

Step 2: Add Lookup Table File

Lookup table files → + Add new
Select destination app (search, SA-IdentityManagement, or custom app)
Upload your CSV file
Name the lookup (e.g., privileged_users)

Step 3: Create Lookup Definition

Lookup definitions → + Add new
Name: privileged_users (must match what you use in SPL)
Type: File-based
Lookup file: privileged_users.csv
☑️ Configure time-based lookup (if time field exists)

Step 4: (Optional) Create Automatic Lookup

Automatic lookups → + Add new
This auto-enriches events at search time without explicit | lookup command

Method 2: REST API (For Automation)

Upload CSV via REST API
# Upload lookup file
curl -k -u admin:password \
  https://your-splunk-cloud:8089/servicesNS/nobody/search/data/lookup-table-files \
  -F "name=privileged_users.csv" \
  -F "eai:data=@/path/to/privileged_users.csv"

# Create lookup definition
curl -k -u admin:password \
  https://your-splunk-cloud:8089/servicesNS/nobody/search/data/transforms/lookups \
  -d "name=privileged_users" \
  -d "filename=privileged_users.csv"

Scheduled Search to Generate Baseline
``` Run daily at 2 AM, covers last 30 days
index=azure sourcetype=azure:aad:signin earliest=-30d latest=-1d
| spath
| eval user = 'properties.userPrincipalName'
| eval country = 'properties.location.countryOrRegion'
| eval city = 'properties.location.city'
| eval hour = strftime(_time, "%H")
| eval device = 'properties.deviceDetail.deviceId'
| where 'properties.status.errorCode' = 0
| stats 
    values(country) as typical_countries
    values(city) as typical_cities
    values(hour) as typical_hours
    values(device) as typical_devices
    BY user
| eval last_updated = strftime(now(), "%Y-%m-%d")
| outputlookup user_baseline.csv

Lookup Troubleshooting

IssueDiagnosticSolution
Lookup returns no matches | inputlookup privileged_users.csv Verify file exists and has data
Case mismatch Check if user vs User Set case_sensitive_match = false
Field name mismatch Compare CSV header to SPL field Use AS keyword: lookup X field AS alias
CIDR not matching Verify CIDR notation (10.0.0.0/8) Ensure match_type = CIDR(field) in transforms.conf
Wildcard not working Check for * in lookup data Use match_type = WILDCARD(field)
Lookup slows search Check lookup file size Move to KV Store if > 100MB
OUTPUT vs OUTPUTNEW Fields being overwritten Use OUTPUTNEW to preserve existing values
Lookup Debugging Queries
``` Test lookup file exists and has data
| inputlookup privileged_users.csv
| stats count

``` Test specific match
| makeresults 
| eval user = "john.admin@myfirstpro.com"
| lookup privileged_users userPrincipalName AS user OUTPUT is_privileged role
| table user is_privileged role

``` Debug why lookup isn't matching
index=cyberark sourcetype=cyberark:identity:audit earliest=-1h
| head 10
| eval user = ActorId
| lookup privileged_users userPrincipalName AS user OUTPUT is_privileged
| table user is_privileged
| where isnull(is_privileged)
``` If is_privileged is null, the lookup didn't match

``` Check for field value differences
| inputlookup privileged_users.csv
| eval user_lower = lower(userPrincipalName)
| table userPrincipalName user_lower

KV Store Lookups (Advanced)

When to Use KV Store
Use KV Store when: data updates frequently (daily), data is > 100MB, you need CRUD operations via REST API, or you need atomic updates.
collections.conf - Define KV Store Collection
[threat_intel_kv]
enforceTypes = true
field.ip_address = string
field.threat_type = string
field.confidence = string
field.source = string
field.first_seen = time
field.last_seen = time
accelerated_fields.ip = {"ip_address": 1}
transforms.conf - KV Store Lookup Definition
[threat_intel_kv_lookup]
external_type = kvstore
collection = threat_intel_kv
fields_list = ip_address, threat_type, confidence, source, first_seen, last_seen
case_sensitive_match = false
KV Store CRUD Operations
``` Read all records
| inputlookup threat_intel_kv_lookup

``` Add/Update records via SPL
| makeresults 
| eval ip_address="192.0.2.100", threat_type="botnet", confidence="high"
| outputlookup threat_intel_kv_lookup append=true

``` Delete specific record (requires _key)
| inputlookup threat_intel_kv_lookup
| where ip_address="192.0.2.100"
| delete threat_intel_kv_lookup

``` REST API - Add record
curl -k -u admin:password \
  -H "Content-Type: application/json" \
  -d '{"ip_address":"192.0.2.100","threat_type":"botnet"}' \
  https://splunk:8089/servicesNS/nobody/search/storage/collections/data/threat_intel_kv
