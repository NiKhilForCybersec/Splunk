What is Splunk?

Simple Explanation

Splunk is a log management and SIEM platform. It collects logs from various sources (servers, firewalls, applications, cloud services), stores them in a searchable format, and lets you search, analyze, visualize, and alert on that data.

Think of it as Google for your logs — but with superpowers. You can search across billions of events, create dashboards, and automatically alert when something suspicious happens.

How to Explain It (If Asked)

"Splunk is an enterprise SIEM and log analytics platform. It ingests 
machine data from across the infrastructure, normalizes it using the 
Common Information Model, and provides real-time search, correlation, 
and alerting capabilities. For this project, I'm using it to build 
detection logic against WAF and identity data sources."

Core Concepts (Memorize These)

TermWhat It IsReal ExampleIf Asked, Say...
Index A container/bucket where data is stored. Like a database table. index=akamai — all Akamai WAF logs "An index is Splunk's storage container for a specific data type. We separate indexes by data source for retention management and access control."
Sourcetype The format/type of data. Tells Splunk how to parse it. sourcetype=akamai:siem "The sourcetype defines the data format so Splunk knows how to extract fields. It's critical for proper parsing."
Source Where the data came from (file path, input name). source=http:akamai_hec "Source identifies the input path. Useful for troubleshooting data ingestion issues."
Host The machine that generated the event. host=web-server-01 "Host identifies the originating system. Important for correlation and troubleshooting."
Event A single log entry/record in Splunk. One line of a log file "An event is a single record — typically one log line or JSON object."
Field A key-value pair extracted from an event. src_ip=192.168.1.1 "Fields are extracted values from events that we can search and analyze. They can be extracted at index time or search time."
_time The timestamp of when the event occurred. Every event has _time "_time is the event timestamp. Accurate timestamps are critical — without them, we can't correlate or trend."
_raw The original, unparsed event text. The raw log line "_raw is the original event before field extraction. Useful when troubleshooting parsing issues."
_indextime When Splunk received/indexed the event. May differ from _time "_indextime is when Splunk ingested the event, which may differ from event time. Large gaps indicate ingestion delays."

Search Basics (SPL)

SPL = Search Processing Language. This is how you query Splunk.

Search Structure

Basic Search Pattern
index=  |  |  | ...

Example:
index=akamai action=deny | stats count by src_ip | sort -count | head 10

Breakdown:
- index=akamai       → WHERE to search (which data)
- action=deny        → FILTER (only denied requests)
- |                  → PIPE (sends results to next command)
- stats count by src_ip → AGGREGATE (count events per IP)
- sort -count        → SORT descending by count
- head 10            → LIMIT to top 10

Essential Commands (You'll Use Daily)

CommandWhat It DoesExample
stats Aggregate data (count, sum, avg, etc.) | stats count by src_ip
table Display specific fields as table | table _time, src_ip, action
where Filter results with conditions | where count > 100
eval Create/modify fields | eval status=if(action="allow","OK","BLOCKED")
rex Extract fields using regex | rex field=uri "(?/api/[^/]+)"
lookup Enrich with external data | lookup privileged_users.csv user OUTPUT is_privileged
timechart Time-based aggregation | timechart span=1h count by action
dedup Remove duplicates | dedup src_ip, user
sort Sort results (- for descending) | sort -count
head/tail First/last N results | head 10
rename Rename fields | rename src_ip as "Source IP"
fillnull Replace null values | fillnull value="unknown"

Real Search Examples (Copy & Modify)

1. Check if Data is Flowing
index=akamai earliest=-1h | stats count
``` If count > 0, data is flowing. If 0, problem.
2. Top 10 Source IPs (Blocked)
index=akamai action=deny earliest=-24h
| stats count by src_ip
| sort -count
| head 10
3. Events Over Time (Hourly)
index=akamai earliest=-24h
| timechart span=1h count by action
4. Find Specific Attack Pattern (SQLi)
index=akamai earliest=-24h
| search rule_tag="SQL*" OR rule_message="*injection*"
| stats count by src_ip, rule_tag
| where count > 5
5. Check Data Latency
index=akamai earliest=-1h
| eval latency_sec = _indextime - _time
| stats avg(latency_sec) as avg_latency, max(latency_sec) as max_latency
| eval avg_latency = round(avg_latency, 0)
``` If avg > 300 (5 min), investigate ingestion delays
6. Use a Lookup to Enrich Data
index=cyberark earliest=-24h
| lookup privileged_users.csv user as ActorId OUTPUT is_privileged, role
| where is_privileged="yes"
| stats count by ActorId, role, EventType

Alerts vs Reports vs Dashboards

TypeWhat It IsWhen to UseHow to Explain
Saved Search A search you save to run again Reusable queries "A saved search is a reusable query. Foundation for reports and alerts."
Report Saved search that runs on schedule, outputs results Daily/weekly summaries "A report is a scheduled search that generates output — CSV, dashboard panel, or email."
Alert Saved search that triggers action when condition met Real-time or near-real-time detection "An alert is a scheduled search that triggers actions when conditions are met — email, ticket, webhook, or notable event in ES."
Dashboard Visual display of multiple searches/panels Monitoring, reporting, presentations "A dashboard is a collection of visualizations for monitoring and reporting. Each panel is powered by a search."
Correlation Search (ES) Alert that creates Notable Events in ES Security detections in Enterprise Security "A correlation search is an ES-specific alert that creates notable events for SOC triage in the Incident Review dashboard."

Common Mistakes to Avoid

MistakeWhy It's BadCorrect Approach
Searching without index= Searches ALL indexes — very slow, may timeout Always specify index=xxx
Using * wildcard liberally Wildcards are expensive, slow searches Be specific: action=deny not action=*deny*
No time range Defaults to 24h but may search too much Always specify earliest=-1h or similar
Filtering after stats Processes more data than needed Filter early: index=x action=deny | stats
Not testing searches before saving Broken alerts, no results Run manually first, verify results
Hardcoding values Inflexible, breaks when things change Use lookups for lists, macros for reusable logic

⚠️ If They Ask Technical Splunk Questions

QuestionExpert Answer
"What's the difference between search-time and index-time extraction?" "Index-time extraction happens when data is ingested — fields are stored in the index. It's faster to search but uses more storage and can't be changed later. Search-time extraction happens when you run a query — more flexible but adds search overhead. For high-volume data, I use index-time for critical fields; for flexibility, search-time."
"How do you optimize a slow search?" "I follow these principles: (1) Filter early — put restrictive terms first, (2) Specify index and time range explicitly, (3) Avoid wildcards when possible, (4) Use tstats for data model accelerated searches, (5) Use summary indexing for frequently-run expensive searches. I can show you the job inspector for specific optimization."
"What's a data model?" "A data model is a hierarchical structure that maps raw data to a normalized schema. When accelerated, it pre-computes summaries for fast searching. In ES, the CIM data models — like Authentication, Web, Network Traffic — let me write detections that work across multiple sourcetypes."
"What's tstats?" "tstats is a high-performance command that searches accelerated data models or tsidx files. It's dramatically faster than regular stats because it uses pre-computed summaries. I use it for high-volume searches against CIM-compliant data."
"How does Splunk Cloud differ from Enterprise?" "Splunk Cloud is the SaaS version — Splunk manages the infrastructure. Key differences: we can't directly edit system configs (props.conf, transforms.conf) — we need custom apps. App installations require vetting (security review). There are also search concurrency and resource limits. But it's fully managed, so no infrastructure maintenance."
